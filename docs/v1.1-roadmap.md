## 1. Lepszy prompt do Claude

**Problem:** Niektóre przyciski w Obsidian/Linear/Cursor nie matchują, mimo że LLM-generated reguły istnieją. Diagnoza pokazała że tytuły w AX różnią się od tych które Claude wygenerował z web docs (np. AX: `"open quick switcher"` vs reguła: `"Quick switcher"` — tu match jest, ale dla niektórych przycisków różnica jest większa).

**Co zrobić:**
- Dodać do `buildSystemPrompt()` w `backend/src/prompt.ts` instrukcję: "Each rule's `titles` array MUST include 3-5 variants: the English label, the verb-led form (e.g. `Open Quick Switcher`), the noun-only form (e.g. `Quick Switcher`), and common verbose forms (e.g. `quick switcher button`)".
- Dodać kilka konkretnych przykładów multi-title (few-shot examples) w prompcie.
- Rozważyć osobny "verify" przebieg: po wygenerowaniu reguł, drugi call do Claude'a: "given this AX skeleton, sprawdź czy każda reguła ma tytuł który matchuje jakiś element w skeleton; jeśli nie — popraw lub usuń".
- Redeploy backendu, ponowny seed dla 4 zweryfikowanych apek.

**Mierzalny efekt:** ręcznie spróbuj 10 przycisków per apka, policz % toastów. Cel: >70% (obecnie ~50%).

## 2. Feedback loop (Phase F)

Zaprojektowane już w spec: `docs/superpowers/specs/2026-05-11-sflow-llm-rules-design.md` sekcja 7 + 11. Tasks F1-F4 w planie: `docs/superpowers/plans/2026-05-11-sflow-llm-rules.md`.

**Skrót implementacji:**
- Backend `/v1/feedback` endpoint — agreguje reporty `{bundleId, ruleIndex, type}` w KV z TTL 30 dni
- Przy ≥5 unikalnych IP zgłaszających ten sam rule jako wrong → auto-disable globalnie + trigger re-discovery z updated promptem ("rule X for action Y was reported wrong, generate alternative")
- Client: cmd-klik na toast (lub corner X) → POST do `/v1/feedback`
- Local override: `user_overrides.json` (już zaprojektowane w RuleCache) — user może na stałe nadpisać złą regułę swoją

**Co to daje:** rules database self-improves w czasie. Im więcej userów, tym szybciej martwe reguły znikają.

**Risk:** trolling (jeden user zgłasza w kółko). Mitigacja: 1 zgłoszenie per `{IP, ruleIndex}` w oknie 30 dni, hard cap.

## 3. Pro tier BYOK (Phase E)

Zaprojektowane w spec sekcja 10. Tasks E1-E4 w planie.

**Założenie biznesowe:** $10 surcharge nad base $25, albo "free perk for early adopters". Decyzja przy launchu.

**Co dostaje Pro user:**
- Pole "Anthropic API key" w Settings (przechowywane w macOS Keychain via `Security.framework`)
- Toggle "Disable all telemetry" — nawet anonimowe `/v1/discover` zapytania nie idą przez nasz backend
- Jak klucz wypełniony → `DiscoveryClient` konstruuje to samo zapytanie ale strzela do Anthropic bezpośrednio (nie do CF Workera)
- Web_search tool nadal działa (Anthropic obsługuje go server-side w SDK)
- Koszt ~$0.05 per nowa apka na ich kluczu (mniej niż 1 piwo miesięcznie nawet dla heavy usera)

**Implementacja:**
- Nowa klasa `KeychainStore` (10-30 linii) — wrapper na `SecKeychain` API
- `DiscoveryClient` ma optional `byokKey: String?` — jeśli jest, route bezpośrednio do `https://api.anthropic.com/v1/messages`
- Settings window (nowy SwiftUI view) z togglem i polem klucza
- Update privacy docs żeby wymienić Pro tier opt-out

**Target customer:** enterprise userzy z compliance, power-userzy którzy nie ufają proxy.

## 4. Lepsza wymiana tytułów na podstawie missów

**Problem długoterminowy:** apki updatują UI, tytuły przycisków się zmieniają, reguły się starzeją. Bez mechanizmu odświeżania, jakość degrades.

**Pomysł:**
- Klient lokalnie loguje "miss events": gdy klik na element jest w "interaktywnej roli" ALE żadna warstwa (L0.5-L4) nie zwróciła reguły → zapisz `{bundleId, role, title, desc, help, timestamp}` do `~/Library/Application Support/SFlow/misses.jsonl`
- Co 24h (background timer): jeśli liczba missów dla apki X > 20 i są tam powtarzające się tytuły (>=3 razy ten sam) → POST do nowego `/v1/refresh` z misses jako kontekst
- Backend: prosi Claude'a "given current rules for X, and these unmatched elements: [...], generate updated rules array" → invalidate cache, zapisz nową wersję, klient pulluje
- Globalnie agregowane: jeśli 50 różnych userów miss'uje ten sam button, system uczy się szybko

**Privacy considerations:** misses zawierają tytuły AX (te same które już wysyłamy w discovery). Te same zasady filtrowania (skeleton) muszą obowiązywać.

**Implementacja:**
- Nowy plik `MissLogger.swift` — zapisuje JSON lines
- Nowy timer w `AppDelegate` (background dispatch queue)
- Nowy endpoint `/v1/refresh` na backendzie — bardzo podobny do `/v1/discover`
- Update prompta Claude'a żeby przyjmował "current rules" + "miss examples"

## 5. Local miss log (`events.jsonl` rozbudowany)

Najlżejsza wersja punktu 4 — tylko lokalna analiza, bez backendu. Dobre na start (nie wymaga deployu).

**Co zrobić:**
- Rozbudować istniejący `EventLogger` (w `SFlow/EventLogger.swift`) żeby logował też MISSES (nie tylko emicje toastów)
- Format JSONL w `~/Library/Application Support/SFlow/events.jsonl`:
  ```json
  {"type":"toast","ts":"2026-05-13T17:55:00Z","bundleId":"com.tinyspeck.slackmacgap","keys":["meta","k"],"source":"bundled"}
  {"type":"miss","ts":"2026-05-13T17:55:30Z","bundleId":"md.obsidian","role":"AXGroup","desc":"some button","title":""}
  ```
- Dodać prosty CLI flag `SFlow --analyze` który czyta `events.jsonl` i printuje statystyki: top apki z toastami, top apki z missami, najczęstsze AX desc/title pary bez toastu

**Co to daje:** Filip widzi gołym okiem które apki są problematyczne, które przyciski "uciekają" — informuje priorytety dla v1.1 punktu 1 lub 4.

**Privacy:** lokalne, nigdy nie opuszcza Maca. Można też dodać "Clear logs" button w Settings.

Gdy wrócisz do tego za tydzień / miesiąc:

1. **Najpierw #5** (miss log) — daje dane do podjęcia decyzji o reszcie. Małe ryzyko, lokalne, ~2h pracy.
